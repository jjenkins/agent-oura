**Arg parsing (per D-01, D-02):**
Parse `process.argv.slice(2)` manually with a simple `getArg(name)` helper that finds `--name` and returns the next arg. Support these flags:
- `--endpoint` (string): single endpoint name (e.g., `daily_sleep`)
- `--endpoints` (string): comma-separated endpoint names for multi-endpoint mode
- `--correlate` (string): comma-separated pair for correlation mode (handled in Plan 02)
- `--start` (string): start date YYYY-MM-DD
- `--end` (string): end date YYYY-MM-DD
- `--offset` (number): day offset for correlation (handled in Plan 02)

Mode dispatch: if `--correlate` is present, print `{"error": "Correlation mode not yet implemented"}` and exit 0 (placeholder for Plan 02). If `--endpoints` is present, run multi-endpoint mode. If `--endpoint` is present, run single-endpoint mode. If none, print `{"error": "Usage: node query.mjs --endpoint <name> [--start YYYY-MM-DD] [--end YYYY-MM-DD]"}` and exit 1.

**Date range logic (per D-07, D-08):**
Implement `getDateRange(startArg, endArg)` that:
- Defaults `end` to today's date via `new Date().toLocaleDateString('en-CA')`
- Defaults `start` to 7 days ago (D-07)
- Computes day difference; if > 60, sets `warning` string: `"Date range (N days) exceeds Oura API ~60-day limit. Results may be incomplete."` (D-08)
- Returns `{ start, end, warning }` (warning is null if <= 60 days)

**Endpoint-aware param builder (per D-10):**
Implement `buildParams(endpoint, startDate, endDate)`:
- If endpoint is `'heartrate'`: return `{ start_datetime: '${startDate}T00:00:00', end_datetime: '${endDate}T23:59:59' }`
- All other endpoints: return `{ start_date: startDate, end_date: endDate }`

**Pagination (per Research Pitfall 2):**
Implement `async function fetchAllPages(endpoint, params)`:
- Loop: call `ouraGetWithRetry('/usercollection/' + endpoint, queryParams)`
- After each call, check `json.next_token`. If present, set `queryParams.next_token = json.next_token` and loop.
- Accumulate `json.data` arrays via concat.
- Return the accumulated array.
- Handle the case where `ouraGetWithRetry` throws -- let it propagate (caller handles).

**Heart rate aggregation (per Research Pitfall 3):**
Implement `function aggregateHeartRateByDay(samples)`:
- Group samples by `sample.timestamp.slice(0, 10)` (YYYY-MM-DD)
- Per day, compute: `bpm_avg` (rounded), `bpm_min`, `bpm_max`, `sample_count`, `sources` (object with source string keys and count values)
- Return array of `{ day, bpm_avg, bpm_min, bpm_max, sample_count, sources }` sorted by day ascending.

**Summary stats computation (per D-14, D-15):**
Implement `function computeStats(records, fieldFn)`:
- Extract values via `records.map(fieldFn).filter(v => v !== null && v !== undefined)`
- If empty, return `{ avg: null, min: null, max: null, trend: null, count: 0 }`
- Compute `avg` (rounded to 1 decimal), `min`, `max`, `count`
- Compute `trend`: split values at midpoint, compare second-half avg to first-half avg. Difference > 2 = `'improving'`, < -2 = `'declining'`, else `'stable'`
- Return `{ avg, min, max, trend, count }`

**Stress value-count summary (per Research Pitfall 5):**
Implement `function stressSummary(records)`:
- Count occurrences of each `day_summary` value (e.g., `{ "restored": 3, "normal": 2, "stressed": 1 }`)
- Return the count object

**Field extraction mapping:**
Create a `FIELD_MAP` object that maps endpoint names to their primary numeric field accessor function:
```javascript
const FIELD_MAP = {
  daily_sleep:     r => r.score,
  daily_readiness: r => r.score,
  daily_activity:  r => r.score,
  daily_spo2:      r => r.spo2_percentage?.average,
  // heartrate: handled separately via aggregateHeartRateByDay
  // daily_stress: handled separately via stressSummary
  // workout: no single score -- return records only
  // session: no single score -- return records only
};
```

**Single-endpoint mode:**
1. Call `getDateRange(startArg, endArg)` for dates
2. Call `buildParams(endpoint, start, end)` for query params
3. Call `fetchAllPages(endpoint, params)` to get records
4. If endpoint is `'heartrate'`: aggregate via `aggregateHeartRateByDay`, then compute stats on aggregated records using `r => r.bpm_avg` as fieldFn. Set `records` to aggregated result.
5. If endpoint is `'daily_stress'`: compute `stressSummary` instead of numeric stats. Set `summary` to the stress count object.
6. If endpoint has an entry in `FIELD_MAP`: compute stats via `computeStats(records, FIELD_MAP[endpoint])`.
7. Otherwise (workout, session): set `summary` to `null` (no numeric summary -- Claude interprets raw records).
8. Output JSON to stdout:
```json
{
  "endpoint": "<endpoint>",
  "start_date": "<start>",
  "end_date": "<end>",
  "warning": null | "<string>",
  "summary": { "avg": ..., "min": ..., "max": ..., "trend": "...", "count": N } | { stress counts } | null,
  "records": [...]
}
```

**Multi-endpoint mode (per D-04):**
1. Split `--endpoints` value by comma
2. Use `Promise.allSettled` to fetch all endpoints in parallel (each via `fetchAllPages` + stats, same as single-endpoint logic)
3. Output JSON to stdout:
```json
{
  "mode": "multi",
  "start_date": "<start>",
  "end_date": "<end>",
  "warning": null | "<string>",
  "results": {
    "<endpoint>": { "summary": {...}, "records": [...] },
    "<endpoint>": { "summary": {...}, "records": [...] },
    "<endpoint>": { "error": "<message>" }
  }
}
```
For rejected promises, include `{ "error": err.message }` for that endpoint.

**Error handling:**
Wrap the entire main flow in a try/catch. On error, output `{"error": "<message>"}` to stdout and exit 1. Use `ouraGetWithRetry` which already produces user-friendly error messages.

**Output (per D-05):**
All output is JSON via `process.stdout.write(JSON.stringify(result, null, 2) + '\n')`. No chalk, no tables, no plaintext formatting.
